## 🟠 HIGH PRIORITY — Fix Before GSC Push

### 5. Missing JSON-LD Structured Data (All Pages)

**Problem:** No `application/ld+json` script blocks found across the entire site. Google uses structured data for rich snippets, knowledge panel eligibility, and enhanced SERP display.

**Recommended schema types:**
- **Review pages:** `Product` + `Review` schema with `aggregateRating`, `offers`, `brand`
- **Roundup pages:** `ItemList` with `ListItem` schema
- **Comparison pages:** `Review` schema for both products
- **Guides:** `Article` or `BlogPosting` schema with `author`, `datePublished`, `dateModified`
- **Homepage:** `Organization` schema with logo, sameAs links, description

**Action:** Add `<script type="application/ld+json">` to `BaseLayout.astro` or page-specific components, injecting frontmatter data dynamically.

### 6. Images Missing `width`/`height` Attributes (79 pages)

**Problem:** `<img>` tags on 79 pages lack explicit `width` and `height` attributes (or CSS aspect-ratio equivalents). This causes **Cumulative Layout Shift (CLS)**, which is a Core Web Vitals metric and a confirmed Google ranking signal.

**Affected components:** Likely `index.astro` hero image, review page hero images, roundup logos, and any `LegacyRawHtml` rendered images.

**Action:**
- Add `width`/`height` attributes to all `<img>` tags in Astro components
- For responsive images, use `aspect-ratio` CSS or the `sizes` attribute
- For `LegacyRawHtml` content, post-process the HTML string to inject dimensions

## 🟡 MEDIUM PRIORITY — Backlog

### 11. Overly Long Meta Descriptions (9 pages)

Descriptions exceeding ~170 characters may be truncated in SERPs. Examples: `fieldedge-vs-servicetitan` (283 chars), `servicetitan-alternatives` (202 chars). These don't hurt rankings directly but waste SERP real estate.

### 12. Missing `twitter:card` on Google Verification Page

The `google328457cf5bd1a6ea.html` file doesn't have Twitter Card meta. This is trivial — it's a verification file and doesn't need social metadata.

### 13. GA4 Loaded Before Consent (Potential Compliance Issue)

The current implementation loads the GA4 script tag immediately (`<script async>`), then gates `send_page_view` and `analytics_storage` behind `localStorage.getItem('csh_cookie_consent')`. This is technically "loading before consent" — the script itself downloads before the user clicks Accept. For strict GDPR compliance, the script tag itself should be injected dynamically only after consent is given. For CCPA/standard US compliance, the current approach is generally acceptable.

### 14. `/go/*` Links Not Present in `dist/` (Build Gap)

Because `/go/*` redirects only live in `vercel.json`, any future migration to a non-Vercel host would lose these redirects. Consider adding a static `dist/go/index.html` or `_redirects` file for portability.
